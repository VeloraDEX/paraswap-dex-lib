# Go Build Migration: Implementation Details

Last updated: 2026-05-15

This document tracks implementation details for
`docs/plans/go-build-migration/implementation.md`. It is organized by
implementation phase. Phase 1 and Phase 2 are complete; later sections should
be added when the next phase is planned.

## Phase 1: Go Module And Fixture Foundation

This section breaks the first implementation slice into concrete work. It is
intentionally scoped to:

- Phase 1: Go module, fixture contract, manifest, loader, and loader tests.
- Phase 2 foundation interfaces: Go types/dependencies required by later
  encoding phases, without implementing executor bytecode or transaction
  assembly.

### Current Status

- `docs/plans/go-build-migration/implementation.md` defines the high-level Go
  parity plan.
- Phase 1 has added root `go.mod`, `go.sum`, and the initial Go source tree
  under `go/txbuilder/...`.
- Resolved-build fixtures are generated by
  `tests/generic-swap-transaction-builder/fixtures/generate-resolved-build-fixtures.ts`.
- TypeScript validates fixture schema in
  `tests/generic-swap-transaction-builder/fixtures/resolved-build-schema.ts`.
- TypeScript fixture generation emits the committed Go manifest and
  executor-address contract JSON.
- `tests/generic-swap-transaction-builder/fixtures/resolved-build-deps.ts`
  owns the current default executor addresses used by resolved-build tests.

### Execution Rule

Phase 1 should land with runnable Go checks and no transaction-encoding
behavior. If the fixture loader, manifest, or foundation interfaces need churn
after executor implementation starts, fix this plan first and keep the Phase 1
boundary small.

Do not implement `BuildTransactionFromResolved`, executor bytecode builders,
ABI calldata assembly, direct builder parity, or a TypeScript-to-Go runtime
bridge in this phase.

### In Scope

- Add root Go module files: `go.mod` and `go.sum`.
- Add Go packages under `go/txbuilder/...`.
- Add a TypeScript-generated resolved-build manifest committed next to the
  fixture root.
- Add a TypeScript-generated executor-address contract JSON committed next to
  the fixture helpers.
- Add a Go fixture loader that reads committed fixture JSON directly and
  classifies fixtures into phase buckets.
- Add Go tests that validate loader behavior, fixture metadata, manifest
  coverage, scoped fixture counts/names, and fixture content hashes.
- Add Phase 2 foundation interfaces in Go:
  - resolved input/output DTO shapes needed by the fixture loader and later
    encoding phases
  - `BuildDeps`
  - `EncodingContext`
  - executor address/config types
  - Augustus V6 ABI embedding/parsing helper
  - test-only deps helper used by future Go fixture tests
- Add `go test ./go/...` and `go vet ./go/...` as runnable gates from day one.

### Out Of Scope

- No executor bytecode encoding.
- No generic Augustus calldata encoding.
- No `expectedParams` or `expectedTx` parity assertions yet.
- No negative `expectedError` parity execution yet.
- No direct-build implementation.
- No DEX-specific `getDexParam` or direct DEX encoder work.
- No runtime TypeScript call path changes.
- No npm packaging or binary distribution changes unless Go files unexpectedly
  enter the published npm package.

### TypeScript Fixture Manifest

Add a manifest generated by the same fixture-generation path as the committed
resolved-build JSON. The manifest must be committed and verified by
`yarn fixtures:check`.

Use an adjacent path outside the fixture JSON walk:

```text
tests/generic-swap-transaction-builder/fixtures/resolved-build-manifest.json
```

Do not place the manifest under `resolved-build/`; the Go loader walks every
fixture `.json` under that directory and should not need to special-case a
non-fixture JSON file.

Recommended manifest shape:

```json
{
  "schemaVersion": 1,
  "buckets": {
    "phase2GenericSuccess": [
      {
        "kind": "generic",
        "name": "executor01-simple-sell-approved",
        "path": "generic/executor01-simple-sell-approved.json",
        "sha256": "<raw fixture file sha256>"
      }
    ],
    "phase2GenericNegative": [],
    "phase3DirectSuccess": [],
    "phase3DirectNegative": []
  }
}
```

Manifest rules:

- The manifest's `schemaVersion` is the manifest-format version, independent of
  each fixture's `schemaVersion`. Phase 1 supports only version `1` for both.
- Go manifest loading rejects unsupported manifest `schemaVersion` values, and
  loader tests must cover that rejection.
- `path` is relative to
  `tests/generic-swap-transaction-builder/fixtures/resolved-build/`.
- `name` must match the fixture's `name` field and the file basename.
- `kind` must match the fixture's `kind` field.
- `sha256` is computed by reading each fixture file's raw bytes from disk after
  `writeResolvedBuildFixtures()` has flushed the fixture files. Do not hash
  in-memory fixture objects.
- Success fixtures are bucketed by `kind`.
- Negative fixture bucketing is owned by the generated manifest. For Phase 1,
  the TypeScript manifest generator may derive the bucket from
  `input.routePlan` presence: `routePlan` present means generic; otherwise
  direct.
- If a future generic negative fixture intentionally omits `routePlan` to test a
  "route plan is required" validation rule, classification must move to an
  explicit fixture or manifest field before that fixture lands.
- Manifest entries are sorted by relative path for deterministic diffs.
- The manifest is generated from TypeScript fixture data, not maintained by Go.
- Go tests fail if any walked fixture path or hash differs from the committed
  manifest.

Implementation notes:

- Extend `resolved-build-loader.ts` or add a small
  `resolved-build-go-contracts.ts` helper used by
  `generate-resolved-build-fixtures.ts`.
- Update `fixtures:check` so `git diff --exit-code` and the untracked-file
  check include both `resolved-build-manifest.json` and
  `resolved-build-deps-contract.json`. Both files must regenerate
  deterministically from `yarn fixtures:generate`.
- Keep TypeScript as the authoritative schema validator; Go uses the manifest
  only to detect skipped or changed fixture files.

### Go Fixture Loader Design

The loader package should live under:

```text
go/txbuilder/internal/testfixtures
```

Responsibilities:

- Discover the repo root by walking upward until `go.mod` is found.
- Use `filepath.WalkDir` to recursively load every `.json` fixture under
  `tests/generic-swap-transaction-builder/fixtures/resolved-build/`.
- Parse only the contract fields needed for Phase 1:
  - `schemaVersion`
  - `name`
  - `kind`
  - `coverage`
  - `input`
  - `expectedParams`
  - `expectedTx`
  - `expectedError`
- Required common fields for all fixture kinds are `schemaVersion`, `name`,
  `kind`, `coverage`, and `input`.
- Success fixtures (`generic` and `direct`) additionally require
  `expectedParams` and `expectedTx`.
- Negative fixtures additionally require `expectedError`.
- Valid `kind` values are `generic`, `direct`, and `negative`; unknown values
  fail loading.
- `name` must equal the fixture file basename without the `.json` extension.
  Fixture names must be globally unique across all buckets, so
  `generic/foo.json` and `direct/foo.json` are an invariant violation.
- Ignore unknown fields, including `orchestration`, `boundaryOnly`, and
  `boundaryOnlyReason`.
- Preserve raw fixture bytes for hash checks.
- Keep `input`, `expectedParams`, and unknown nested payloads available as
  `json.RawMessage` until later phases need typed encoding.
- Parse `expectedTx`, `expectedParams`, and `expectedError` as raw fields and
  enforce only the kind-specific presence rules above. Deep structure and
  absence-of-extra-field rules remain owned by the TypeScript schema and
  verified by `yarn fixtures:check`.
- Classify each fixture into one scope bucket:
  - `Phase2GenericSuccess`
  - `Phase2GenericNegative`
  - `Phase3DirectSuccess`
  - `Phase3DirectNegative`
- Validate manifest bucket assignments independently from the manifest entry:
  success buckets must match fixture `kind` and directory, and negative buckets
  must match `input.routePlan` presence. Cross-bucket leakage fails loading.
- Fail on unsupported `schemaVersion`, missing required top-level fields,
  empty `coverage`, invalid `kind`, duplicate fixture names, duplicate relative
  paths, manifest `name`/`kind` mismatch, or hash mismatch.

The loader should not validate the full TypeScript fixture schema. That remains
owned by Jest and `fixtures:check`.

### Acceptance Criteria

Phase 1 is complete when:

- A clean checkout can run `go test ./go/...` successfully.
- A clean checkout can run `go vet ./go/...` successfully.
- Go fixture tests fail if a fixture is skipped, reclassified, or hash-changed
  without regenerating the TS manifest.
- Go fixture tests fail if a manifest entry's `name` or `kind` disagrees with
  the fixture file.
- Go fixture tests load at least one known generic fixture and assert non-empty
  `coverage`.
- Go loader/helper tests reject unsupported manifest and deps schema versions.
- Go fixture tests prove unknown TS-only metadata is ignored.
- `resolved-build-manifest.json` and `resolved-build-deps-contract.json` are
  generated by TypeScript and checked by `yarn fixtures:check`.
- The embedded Go Augustus V6 ABI copy is tested against
  `src/abi/augustus-v6/ABI.json`.
- The Go foundation interfaces compile but expose no transaction-building
  behavior yet.

## Phase 2: Encoding Foundation

Phase 2 implements the reusable encoding primitives and dependency validation
that later `BuildTransactionFromResolved` work depends on. It intentionally
does not add `BuildTransactionFromResolved`, executor bytecode encoding, or
fixture-level `expectedTx` parity yet.

### Current Status

- Phase 1 added Go DTOs, `BuildDeps`, `EncodingContext`, embedded Augustus V6
  ABI loading, fixture loading, and the test-only deps helper.
- Phase 2 added generic Augustus V6 params, fee packing, UUID/block metadata
  packing, tx value derivation, selector checks, and dependency consistency
  validation.
- `BuildInput.RoutePlan`, `BuildInput.ResolvedLegs`, `BuildInput.WethPlan`,
  and direct `Params` still defer semantic parsing with raw JSON fields.
  `BuildInput.Fee` remains raw at the DTO boundary and is parsed by
  `ParseFeeInput`.
- Go still has not implemented `BuildTransactionFromResolved`, executor
  bytecode encoding, direct builder behavior, or fixture-level `expectedTx`
  calldata parity.
- The canonical TypeScript source for Phase 2 behavior is
  `src/generic-swap-transaction-builder/resolved/build-transaction.ts`.

### Goal

Land the smallest Go encoding foundation that Phase 2a can call directly:

- typed generic fee and gas DTOs
- dependency/context validation helpers
- Augustus V6 method selectors
- UUID/block metadata packing
- partner-and-fee packing
- generic swap tuple construction without executor bytecode generation
- tx value derivation from source token, side, and amount fields

### In Scope

- Add typed Go DTOs needed by shared encoding primitives:
  - `FeeInput`
  - generic swap tuple metadata shape
  - helper output shape for generic params without bytecode
- Parse `BuildInput.Fee` into `FeeInput`.

  - network

  - wrapped native token address

### Fee Packing

Implement fee packing with `math/big`. Do not use fixed-width integers. Keep
the same two-layer structure as TypeScript so the public `FeeInput` mapping is
separate from bit packing.

Layer 1, `BuildFeesV6`, maps public fixture input to internal packer fields:

- If `fee.ReferrerAddress != nil`:
  - `partner = *fee.ReferrerAddress`
  - `feePercent = "0"`
  - `isReferral = true`
- Otherwise:
  - `partner = fee.PartnerAddress`
  - `feePercent = fee.PartnerFeePercent`
  - `isReferral = false`
- All other booleans pass through unchanged.

Layer 2, `packPartnerAndFeeData`, packs those internal fields:

Parity rules:

- `partner = (feePercent == "0" && !isTakeSurplus && !isReferral) ? 0 : partner`
- high 160 bits are `partner << 96`
- if `feePercent != "0"`, set `feePercent & 0x3FFF`
- else if `isTakeSurplus`, set `IS_TAKE_SURPLUS_MASK = 1 << 95`
- else if `isReferral`, set `IS_REFERRAL_MASK = 1 << 94`
- if `isSkipBlacklist`, set `IS_SKIP_BLACKLIST_MASK = 1 << 93`
- if `isCapSurplus`, set `IS_CAP_SURPLUS_MASK = 1 << 92`
- if `isDirectFeeTransfer`, set `IS_DIRECT_TRANSFER_MASK = 1 << 91`
- if `isSurplusToUser`, set `IS_USER_SURPLUS_MASK = 1 << 90`
- Return the decimal string representation used in `expectedParams[2]`.

Acceptance tests:

- Use representative committed generic fixtures:
  - zero-fee fixture: `executor01-simple-sell-approved`
  - nonzero partner fee: `fee-nonzero-partner`
  - referrer: `fee-referrer`
  - take surplus: `fee-take-surplus`
  - surplus to user: `fee-surplus-to-user`
  - direct transfer: `fee-direct-transfer`
- Each test compares the Go decimal string to fixture `expectedParams[2]`.
- Add one synthetic Go unit vector for `isSkipBlacklist` until a committed
  fixture covers it. The expected value must be computed from the documented
  TypeScript masks, including `IS_SKIP_BLACKLIST_MASK = 1 << 93`, so a broken
  skip-blacklist bit cannot pass Phase 2.
